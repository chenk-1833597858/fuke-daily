# 人间非虚构冲突 · 采集与挖掘

> 网易「人间theLivings」是中国最大的非虚构叙事平台，提供了大量真实的中国本土冲突案例。
> 与虚构故事不同，人间文章中的冲突细节（因/缘）是真实发生的——当你需要"真实的因果链"作为写作素材时，人间是最可靠的来源。

## 文章保存技巧

### 域名规则

| 域名 | curl 行为 | 解决方法 |
|------|----------|---------|
| `m.163.com/dy/article/...` | ✅ 正常下载 | 直接 curl 即可 |
| `m.163.com/exclusive/article/...` | ✅ 正常下载 | 被删文章可能迁移到这里 |
| `www.163.com/dy/article/...` | ❌ 403 Forbidden | 加移动端 User-Agent |
| `m.163.com/renjian/article/...` | ❌ 403 Forbidden | 加移动端 User-Agent |
| `www.163.com/renjian/article/...` | ❌ 403 Forbidden | 加移动端 User-Agent |

### 下载命令

```bash
# 对 www.163.com / renjian.163.com 域名，必须加移动端 UA
curl -sL -o "文章名.html" \
  -A "Mozilla/5.0 (Linux; Android 10; K) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/131.0.0.0 Mobile Safari/537.36" \
  "https://www.163.com/dy/article/KP65AHIA0556AKV3.html"

# m.163.com 域名直接下就好
curl -sL -o "文章名.html" "https://m.163.com/dy/article/KMMTFCHG052188CT.html"

# renjian 子域名（www.163.com/renjian/article/...）会 403 甚至 146 字节空文件
# 如果遇到这种情况，尝试更换到 news 子域名也同样 403。可靠方案是用移动端 UA 走原域名：
curl -sL -o "形婚_不愿放弃形婚的女人.html" \
  -A "Mozilla/5.0 (Linux; Android 14) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Mobile Safari/537.36" \
  "https://www.163.com/renjian/article/HPRCIEM9000181RK.html"

# 如果文章 404 了，搜它的标题，可能已迁移到 exclusive/ 路径
```

### 保存目录

**批量下载陷阱：** 连续从同一域名下载多篇文章时，某些文章会返回 403（nginx 限制）。解法：
1. 先尝试无 UA 的 curl，如果只返回 `146 bytes` → 加移动端 UA 重新下载
2. 如果加 UA 还 403 → 用浏览器手动保存
3. `renjian.163.com` 子域名比 `m.163.com` 更常触发 403

## 批量拆解流水线（用户指令模式）

当用户说"都拆出来"、"一篇篇来"、"不用问我直接拆"时，进入**批量拆解流水线模式**：

### 流水线规则

1. **自主执行** — 不需要每篇都问用户确认，直接拆→导入→下一篇
2. **逐篇处理** — 拆完一篇立刻导入向量数据库，再拆下一篇
3. **不中断** — 直到本地所有文章拆完为止
4. **烈度判断** — 每篇拆完后用「是不是XX？」序列从0问到10，第一个"是"即停
5. **敏感过滤** — 遇到政治运动（批斗/文革/上访）、民族宗教冲突等内容直接跳过
6. **去人名化** — 所有人名替换为关系称呼
7. **格式统一** — 每条因果链必须包含：类型、烈度、烈度评分、果、因[]、缘[]

### 流水线执行步骤

```
for 每篇本地未拆文章:
    1. 读取HTML文件
    2. 用BeautifulSoup提取正文
    3. 识别核心冲突（果-因-缘三层）
    4. 判断烈度（0-10）
    5. 去人名化
    6. 构建因果链JSON
    7. 用fastembed生成向量
    8. 导入Qdrant
    9. 验证导入成功
    10. 继续下一篇
```

### 每篇文章的拆解产出

每篇人间非虚构文章通常可提取 **5-8 条** 因果链。拆解时关注：
- **核心冲突**（烈度最高的1-2条）
- **次要冲突**（推动剧情发展的3-4条）
- **背景冲突**（塑造人物关系的2-3条）

### 烈度分布建议

一篇好的人间文章，冲突烈度应该覆盖多个级别：
- 1-2条 8分（害命/极端暴力）
- 2-3条 5-7分（决裂/打架/报复）
- 2-3条 2-4分（顶撞/吵架/拉黑）
- 1-2条 0-1分（日常/应付）

如果一篇文章全是8分或全是2分，说明冲突层次单一，只提取最核心的几条即可。

## 冲突挖掘方法

### 核心流程

从一篇人间非虚构文章中提取冲突链的步骤：

### Step 1：速读抓大结构

读一遍，识别故事的核心冲突：
- 谁是受害者？谁是加害者？
- 争的到底是什么？（表面 vs 实际）
- 冲突是爆发式（一个事件炸了）还是累积式（多年沉默后爆发）？
- 有没有关键转折点？（一句改变一切的台词、一个决定性的行为）

### Step 2：提取因果链（果-因-缘三层）

果是最具体的那个结果——一件可以观察到的事（不是情绪，不是状态）。

**果**：具体发生的事（可观察、可验证）。不要抽象如"她伤透了心"，要具体如"新婚夜被往死里打"或"聚餐时当众掀了桌子"。

**因**：导致这件事发生的直接原因。每个因必须是一个事件或行为，不要写状态。区分：
- 对方的因（他做了什么才引发冲突）
- 自己的因（自己做了什么/没做什么才陷入这个处境）
- 环境的因（什么外部条件让这件事不可避免）

**缘**：让因起作用的偶然条件。缘分三个层次：
- 时间缘（为什么是那个时候发生）
- 关系缘（他们之间有什么特殊关系让冲突升级）
- 社会缘（当时的社会背景/规则让某方无法逃脱）

### Step 3：判断烈度

从0（是不是日常？）问到10（是不是连坐？），第一个"是"就是烈度。

**注意：** 不因为同情受害者就给高分，也不因为事件有暴力就给暴力的分。烈度判断只看冲突本身的对抗程度——双方有没有面对面交锋？有没有选择？有没有不可逆的后果？

### Step 4：去人名化

把所有人名替换为关系称呼（"被打的妻子"、"抛夫弃子的母亲"、"被丈夫背叛的妻子"）。目的是让因果链成为可复用的模板，不绑定具体人物。

### Step 5：标注来源（追溯用）

在入库的因果链 payload 中保留 `source` 字段指向保存的文章文件名。这样以后看到一条因果链可以溯源回原文。

## 注意

1. **文章随时可能被删** — 读到好的立刻下载保存
2. **不编造因果链** — 人间文章中的每一个细节都是真实发生的，提取时不要为了"更有冲突感"而添加原文没有的内容
3. **不代替用户判断烈度** — 只提取果/因/缘，由用户自己决定烈度评分
4. **入库规则：真实冲突直接导入，拆书/小说才问** — 从人间/互联网找到的真实冲突直接提取入库，不用问用户。只有从虚构作品（小说拆书）提取的因果链才需要先预览后入库
5. **敏感内容检查** — 提取后检查是否有：政治运动（批斗/文革/上访）、民族宗教冲突、不符合常识的编造内容。发现此类直接跳过不收录。用户明确说过「批斗的那个不要，敏感话题」
6. **入库前检查烈度评分是否合理** — 用「是不是XX？」逐级判断，不要自己编分。如果烈度评分和果的事实程度不匹配（如打耳光写成害命=8分），说明判断错了
7. **模型缓存被删后处理** — fastembed 模型缓存（`~/.cache/fastembed/`）可能被系统清理。如果搜索时报模型找不到，检查缓存是否存在，不存在时尝试用 sentence-transformers 替代或等待网络恢复后重新下载
